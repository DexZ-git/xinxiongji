## 1. 产品概述

- **名称**：新熊记（承熊熊币的熊系品牌）。
- **使用者**：一个人用（本人）；太太可能也各自装一个。**按单人用设计**。一台手机、不换设备。
- **目标**：把现在散在「备忘录」里的一堆菜，变成一台 iPhone 上「像原生 app」的工具——能定菜单、能自动备菜、能边做边看。
- **规模**：目前十几道，预计扩展到二三十道。够小，不需要复杂搜索/分页。
- **定位/理念**（沿用熊熊币）：
  - **本地优先（local-first）**：数据只存本机浏览器，不上后端、不上云、不外传。
  - **PWA 优先**（非原生 SwiftUI）：开发者在 Windows、无 Mac；复用代码；将来 Capacitor→App Store 的地基。

### 1.1 三大场景（并重，不是只有备料表）
1. **归档菜谱**（场景 1）：按固定一级分类存"我会做的菜"，每道 = 菜名 + 分组步骤 + 一张成品图。
2. **定菜单**（场景 2·仪式感）：像餐厅扫码点单那样"浏览 → 加入菜单"，可让太太/朋友帮选。菜单**存档**留作念想，可手动上传一张**合照当头图**。
3. 菜单落地为**三视图**：① 菜品清单 ② **聚合备料/买菜清单**（核心）③ 烹饪步骤（做菜时轻便查看）。

### 1.2 用户原始痛点（设计依据，别丢）
- 定菜单时没有可视化的地方，常要翻备忘录、还得让别人来选 → 场景 2 的浏览/点单要解决。
- 选定后要**手动从备忘录抽出那几道菜的文本**，很麻烦 → 菜单自动收束这几道菜。
- 备菜靠**短时记忆**记要买/要切什么 → 聚合备料表替代记忆。
- 做菜时要翻抽出来的文本、自己找信息 → 烹饪视图 tab 展开、随时查。

## 2. 技术架构

- **纯前端**：`index.html` + `app.js`（单文件，无框架）+ `styles.css`。无构建步骤、无依赖（同熊熊币）。
- **存储**：
  - **图片 → IndexedDB**（成品图、菜单合照都是 Blob，localStorage 装不下）。以 `recipeId`/`menuId` 为 key 存 Blob，读取时 `URL.createObjectURL`。
  - **结构化数据（菜谱/备料库/菜单）→ localStorage**（键 `xiongji_data_v1`），量小、便于导入导出备份。首次加载用内置 `data/recipes.json` + `data/ingredients.json` 作为种子。
- **离线**：`sw.js`（service worker，缓存外壳，断网可开）；`manifest.json`（display: standalone，名称"新熊记"）。
- **云同步（可选，v0.4）**：**Cloudflare Pages Function `/api/data`** 代理 **jsonbin**（`functions/api/data.js`）。
  - **同步「菜单」+「自建菜谱」文本**（`{menus, userRecipes, updatedAt}`）；照片不上云（二进制、jsonbin 装不下；内置菜图随仓库、自建菜图仅本机）。
  - **单人云备份**：`updatedAt` 时间戳后写覆盖，不做多端合并。**自动**：改动经 `mutate()` 打时间戳 + 防抖(1.5s)上传；打开时 `cloudSync()` 按时间戳对账（云更新才覆盖本地）。
  - **安全**：jsonbin Master Key 只存 CF 环境变量（`JSONBIN_KEY`/`JSONBIN_BIN`），绝不进前端；`/api/data` 用 `SYNC_TOKEN` 校验（用户在「更多」粘贴、存本机 localStorage、不进代码）。配置见 `DEPLOY.md`。
  - 无口令 / 未部署 / 离线时，同步静默跳过，app 照常纯本地可用。SW 不缓存 `/api/`。
- **访问门禁（v0.5）**：打开 app 先过口令墙——**口令 = SYNC_TOKEN**（一码两用：进 app + 开同步）。boot 时探 `/api/data`：返回 **401**（已设口令）才显示门禁，输入口令经 `/api/data` 校验（200=对、401=错）；本地(404)/未配置(500)/离线则直接进。解锁标记存 `xiongji_unlocked`（本机一次、支持离线）。「更多」有「🔒 锁定本机」清除口令。**注意**：这是**前端门禁**，只挡随手打开网址的人；静态文件（recipes.json 等）仍可被直连 URL 取到——要真·私密请上 **Cloudflare Access（Zero Trust 边缘门禁）**。
- **部署**：静态 HTTPS 托管（Cloudflare Pages，连 GitHub 私有仓库自动部署，输出目录 `/`，`functions/` 自动识别）。菜谱不敏感、**无需加密**。步骤见 `DEPLOY.md`。

## 3. 数据模型

三张表：**recipes（菜谱）/ ingredients（备料库）/ menus（菜单）**。菜谱与备料库随代码内置（种子）；菜单是用户运行时创建。

### 3.1 固定一级分类（5 类，已定）
`海鲜 / 肉禽 / 蔬菜 / 汤羹 / 凉菜`。每道菜**归且仅归一类**（不做多归属）。分类固定、暂不支持用户增删（Phase 2 再说）。

### 3.2 备料库 ingredients（`data/ingredients.json`）
把"蒜/姜/葱/螃蟹/生抽…"做成**唯一实体**，菜谱引用它 + 标注形态，这样才能聚合。字段：

| 字段 | 说明 |
|---|---|
| `id` | 拼音短 id，如 `garlic`、`crab` |
| `name` | 显示名，如 `蒜` |
| `group` | 采购分组，三选一：**`生鲜主料` / `香料配菜` / `常备调料`** |
| `emoji` | 可选，列表小图标 |

- **`生鲜主料`**：那条鱼、那块肉、那把菜——**每次都要买**。
- **`香料配菜`**：蒜姜葱、红葱头、干辣椒、小米椒、黄油、芝麻等——要买、且**形态是关键信息**。
- **`常备调料`**：酱油/盐/糖/淀粉/耗油/料酒等——家里常备，买菜清单里**默认折叠、不勾选**。

### 3.3 菜谱 recipes（`data/recipes.json`）

| 字段 | 说明 |
|---|---|
| `id` | 拼音短 id，如 `jiang-cong-chao-xie` |
| `name` | 菜名，如 `姜葱炒蟹` |
| `category` | 5 类之一 |
| `photo` | 成品图（一张）；实际 Blob 存 IndexedDB，此处存 key 或 null |
| `main` | 生鲜主料数组：`[{ing, form?, note?}]`（form 可空，note 如"整只处理·剁尖拍腿"）|
| `aromatics` | 香料配菜数组：`[{ing, form, note?}]`（**form 是核心**：片/粒/蓉/末/段/整…）|
| `seasonings` | 常备调料数组：`[ing, …]`（只列 id，用量不较真）|
| `steps` | 分组步骤：`[{group, items:[…]}]`（group 如"备菜"/"炒"/"制作"，忠实沿用备忘录的分段）|

> **用量约定（已定）**：**不较真**。备料表只记两件关键信息——**哪道菜需要 + 什么形态**。不存克/瓣/根等数量。份量固定（不按人数自动缩放，自己心里换算）。

### 3.4 菜单 menus（运行时创建，存 localStorage）

| 字段 | 说明 |
|---|---|
| `id` | 时间戳/uuid |
| `name` | 菜单名，如"年夜饭"（可留空自动取日期）|
| `date` | 创建/用餐日期 |
| `cover` | 合照头图 Blob key（可选，手动上传，可不传）|
| `recipeIds` | 选中的菜品 id 数组 |

菜单的三视图（备料聚合、菜品清单、烹饪步骤）都是**从 `recipeIds` 实时派生**，不冗余存储。

## 4. 备料聚合口径（核心，最重要）

给定一个菜单的 `recipeIds`，聚合成买菜/备料一览表：

1. **收集**：遍历选中菜谱的 `main` + `aromatics` + `seasonings`，按 `ing` 归并。
2. **按食材聚合、并列形态与菜名**：同一食材若多道菜用到不同形态，全部并列。
   - 例（选了 姜葱炒蟹 + 蒜蓉通菜 + 凉拌菠菜）：
     > **🧄 蒜** — 片《姜葱炒蟹》· 蓉/粒《蒜蓉通菜》· 末《凉拌菠菜》
   - 让用户一眼知道**要切几种形态、各供哪道菜**，一次备齐、不重复切。
3. **分区折叠展示**（已定）：
   - **要买的**（`生鲜主料` + `香料配菜`）→ **默认展开**，是买菜/备菜主清单。生鲜主料在上、香料配菜在下（香料带形态标注）。
   - **常备调料**（`常备调料`）→ **折叠、默认不勾选**（只是提醒"家里有没有"），不喧宾夺主。
4. **勾选态**：每行可勾（买好了/备好了），纯本地 UI 状态，随菜单走。
5. **酱汁/料汁**：留在各菜自己的"备菜"步骤里（如"1/3碗水+半茶匙盐+…"），**不拆进聚合表**（否则调料噪音太大）；聚合表只管"要买/要切的实物 + 形态"。

## 5. UI 结构（规划）

- **底部 Tab（3 个）**：**菜谱 / 菜单 / 更多**（图标见第 6 节；第 4 个 tab 图标备用）。
- **菜谱 tab（场景 1 + 场景 2 入口合一）**：
  - 顶部 **分类筛选 chips**（全部 + 5 类，sticky 吸顶），点「汤羹」即定位到该类。
  - 图片菜卡网格；**非点菜态**点卡 → 单菜详情（成品图 + 分组步骤 + 用到的备料）。
  - 右上角 **「🧺 点菜」** 进入**点菜态**：同一网格变多选，**按图片选**（选中态：番茄描边 + ✓ 徽章）；底部**浮条**显示「已选 N 道」，可**展开已选清单（文字，可逐条 ✕）**；右下「确认成单」→ **命名**（如「7月11日家宴」「阿明来家玩午餐」，默认给「M月D日家宴」）→ 建单并跳到该菜单。
- **菜单 tab（场景 2/3 落地）**：
  - 列表：已存档菜单卡（合照头图 + 名称 + 日期 + 菜品数）。
  - 点开一桌 → **菜单详情 = 3 个操作大卡**：**🧺 备菜 / 🍳 烹制 / 📷 留念**（不拆成全局 Tab——这三者是针对某一桌的操作，脱离菜单无意义）。
    - **备菜**：聚合备料/买菜清单（分区折叠：要买的生鲜+香料展开、常备调料折叠；每样按食材聚合并列形态+菜名；可勾）。
    - **烹制**：这一餐所有做法，三种轻交互切换——**手风琴**（就地展开多道对照）/ **切菜签**（菜名横滑）/ **逐步勾**（每步打勾划掉）。不做并行时间线。
    - **留念**：合照头图（手动上传，可不传）+ 这一桌的菜（图卡，可点进详情）+ 命名/日期/改名。作念想。
- **导航返回**：从「留念」等处点进菜谱详情，返回回到原处（`state._ret` 记录来源）。

## 7. 录入 / 数据工作流

- **内置菜谱**：用户把菜（菜名+步骤+图）给助手，助手提炼成 `data/recipes.json` + 更新 `data/ingredients.json`，随仓库部署。
- **自助录入（v0.6 已上线）**：更多 → **➕ 添加菜谱**。用户只填 菜名/分类/可选成品图/**做法原文**（备忘录颗粒度），app **本地规则解析**（不联网、非 AI）：
  - **材料提炼** `matchIngredients()`：对照备料库（名称拆分 + `ING_ALIAS` 别名表），**最长别名优先 + 字符占位防串词**（"白胡椒粒"不会被记成"胡椒"）；命中后看紧随 1-2 字提取**形态**（`FORMS`：片/末/蓉/粒/段/花/大段…）；按备料库分组归入 主料/香料/调料。词库没有的词自动跳过（以后由助手补库）。
  - **步骤排版** `parseStepsText()`：短标题行（`1、备菜` / `【备菜】`）开新组，`1）2）`行做条目，无组则归「做法」。
  - 解析后**预览可校对**（chips 点 ✕ 删误识别）→ 保存进 `DB.userRecipes`（id `u<时间戳>`、`user:true`），并入菜谱列表/点菜/备料聚合，**随云同步**。自建菜谱详情页可删除（同时从所有菜单里摘掉）。
- 图片由用户在 app 内上传（自建菜成品图、菜单合照），存 IndexedDB、不上云。

## 9. 关键产品决策（为什么这么做）

1. **PWA 优先、本地优先、无后端**：同熊熊币，隐私最大化、零运维、复用代码。
2. **图片走 IndexedDB**：菜谱天生有图，localStorage 装不下；单设备不换机，无需云同步。
3. **备料库做成受控实体**：否则"蒜"无法跨菜聚合成一览表——聚合表是场景 2 的核心价值。
4. **用量不较真、份量固定**：中餐多"适量"，用户对量有数；关键信息是"哪道菜需要 + 什么形态"。
5. **常备调料分区折叠**：买菜聚焦要买的生鲜/香料，又不丢调料检查项。
6. **烹饪不做并行时间线**：用户明确只要"轻便查看、tab 展开"，不要复杂编排。
7. **助手代录入优先**：用户对网页自助编辑没底，先代劳、验证数据模型，再迭代放开。
